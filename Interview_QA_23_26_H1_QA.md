# Interview Guide — Test Case Generation
## AI-генерация тест-кейсов из пользовательских историй

**Hypothesis file:** `Hypotheses_QA_23_26.md` — Hypothesis H1
**DL entry:** Открыть после того как H4 (test run report) запущен и доверие с командой установлено
**Deadline:** ~2–3 недели после H4 пилота

> ⚠️ **Не начинать без H4.** Этот инструмент требует одобрения тимлида + QA менеджера + IT-безопасности (3 уровня). Без предварительно установленного доверия от H4 — вероятность отказа высокая.

---

## Кого интервьюировать

**Роль:** QA-инженер (не только тимлид) — тот, кто получает user stories от разработчиков и пишет тест-кейсы к ним.

**Нужно:** 3–5 QA-инженеров. После тимлида (H4) — попросить познакомить с командой.

**Почему именно они:** Написание тест-кейсов из user stories — #1 подтверждённая боль QA в банках. State Street 67% сокращение времени. DBS: 20 человеко-дней → 1 день. 4–6 ч на одну сложную user story подтверждено синтетическим CustDev.

**Важно сделать до встречи:**
- [ ] Убедиться что H4 пилот запущен или хотя бы одобрен — это даёт кредит доверия
- [ ] Подготовить: "Вот что уже работает у ваших коллег-аналитиков. Теперь хотим понять — подходит ли это для QA."
- [ ] Ключевая фраза: *"Shadow mode: AI генерирует тест-кейсы, вы проверяете независимо. Если совпадает 7 из 10 — продолжаем."*

## Что нужно подтвердить или опровергнуть

| Допущение | Вопрос для проверки | Green сигнал | Red сигнал |
|---|---|---|---|
| Написание тест-кейсов занимает 4–6 ч на историю | "Сколько времени уходит на тест-кейсы для одной user story?" | >2 ч | <30 мин |
| Это реальный bottleneck спринта | "Разработчики иногда ждут пока QA закончит тест-кейсы?" | "Да, регулярно" | "Нет, мы обычно успеваем" |
| User story достаточно как входные данные | "Из чего вы пишете тест-кейсы — из user story, из доп. общения с разработчиком?" | "В основном из user story" | "Нужно много уточнять у разработчика" |
| Shadow mode принимаем | "Готовы сначала проверять AI-кейсы независимо прежде чем использовать?" | "Да, разумный подход" | "Нет времени на параллельную работу" |
| Тимлид + QA менеджер одобрят | "Кто принимает решение о новом инструменте в тестовом pipeline?" | "Тимлид + QA менеджер" | "+ IT безопасность отдельно" |

## Известные возражения — быть готовым

| Возражение | Откуда | Как отвечать |
|---|---|---|
| "Если AI пропустит критический edge case — баг уйдёт в production" | Синтетический CustDev | "Shadow mode решает это: 4 недели AI работает параллельно с вами. Сравниваем — если AI пропускает систематически, останавливаемся." |
| "Кто несёт ответственность если AI пропустил баг?" | Синтетический CustDev | "Вы. Финальный набор тест-кейсов подписываете вы. AI — ваш помощник, не замена." |
| "Наши user stories написаны плохо — AI не поймёт" | Ожидаемо | "Начнём с тех историй где acceptance criteria чёткие. Если user story плохая — AI это явно покажет, это само по себе ценная обратная связь для разработки." |
| "IT-безопасность должна проверить инструмент" | Ожидаемо | "У нас уже есть одобренный pipeline для внутренних AI-инструментов (DL-0, DL-1, DL-3). Можем передать документацию." |
| "У нас специфичная система — AI не знает бизнес-логику" | Ожидаемо | "Для стандартных acceptance criteria (happy path, negative, boundary) — бизнес-логика не нужна. Для специфических edge cases — вы дополняете вручную." |

## Скрипт интервью (~40 минут)

### Блок 1 — Разогрев (5 мин)
- "Расскажи как устроен твой типичный спринт — что занимает больше всего времени?"
- "Какую часть своей работы считаешь по-настоящему творческой?"

### Блок 2 — Боль (12 мин)
- "Когда ты получаешь user story — с чего начинаешь писать тест-кейсы?"
- "Сколько обычно занимает написать тест-кейсы для одной истории? А для сложной?"
- "Бывает что разработчики ждут пока ты закончишь? Насколько это частая история?"
- "Из чего ты пишешь — из user story хватает или нужно уточнять у разработчика?"
- "Что самое долгое — придумать edge cases, оформить, что-то ещё?"

### Блок 3 — Решение (12 мин)
- "Если бы AI прочитал user story и выдал черновик тест-кейсов — happy path, negative cases, edge cases — ты бы с него начинал?"
- "Как бы ты проверял что AI ничего не пропустил?"
- "Готов ли 4 недели работать так: AI генерирует — ты проверяешь независимо — сравниваем совпадение?"
- "Если AI совпадает с тобой в 7 из 10 случаев — это достаточно чтобы начать использовать?"

### Блок 4 — Стопоры (6 мин)
- "Что точно остановило бы от использования, даже если качество хорошее?"
- "Есть типы историй где AI точно не справится? Как часто такие?"
- "Кто принимает решение о новом инструменте в тестовом pipeline — тимлид, QA менеджер, ещё кто-то?"

### Блок 5 — Закрытие (5 мин)
- "Если бы мы взяли 5 user stories которые вы уже тестировали и показали AI-версию тест-кейсов — ты бы сравнил?"
- "Какой тип историй был бы лучшим для первого теста?"

---

## Критерии решения

**🟢 Зелёный:** 3+ из 5 подтверждают: >2 ч на user story, dev ждёт QA в спринте, готовы к shadow mode, тимлид + QA менеджер = достаточно для пилота.
**🟡 Жёлтый:** Боль есть, но user stories плохо написаны → пивот: сначала улучшить шаблон user story вместе с командой, потом AI-кейсы.
**🔴 Красный:** Тест-кейсы <1 ч на историю ИЛИ разработчики не ждут QA ИЛИ нужен отдельный IT-security review который займёт месяцы.

- [ ] Запросить: 5 уже-протестированных user stories с реальными тест-кейсами — для ретроспективного теста
